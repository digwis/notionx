# 0.5.4 Scaffolder Improvements Design

## Summary

This design tightens the `@notionx/create-nextion-app` scaffolder along three
axes — reliability of the Notion provisioner, richness of the Notion-backed
site settings surface, and consistency of the bundled UI — and ships them
together as a single 0.5.4 patch release.

The Notion-API-2025-09-03 refactor (shipped across 0.5.1 and 0.5.3) made
the scaffolder *correct* against the new schema, but it also exposed a
handful of rough edges that the user has been hitting in real use:

1. The site-settings data source is **re-created on every provision** instead
   of being reused, so re-running `nextion update` (or any re-provision) in
   a real project piles up duplicate data sources under the Notion parent
   page.
2. The site-settings row the scaffolder seeds is **effectively empty** — the
   template only writes 4 fields (name, tagline, description, default
   locale), so the home page falls back to `fallbackSiteConfig` and the
   operator can only edit those four things in Notion. Navigation, footer,
   theme, and SEO are all hard-coded in the generated project and never
   reachable from Notion.
3. The CLI still **asks which UI preset to bundle** even though the `site`
   preset is the right answer for every shipped scaffold (blog, page, etc.).
   This is a dead-end question that adds steps and risks generating a
   project that's missing the components the page builder needs.

0.5.4 closes all three.

A larger rework — splitting each Notion content source into N per-locale
data sources so a project can publish real bilingual content — is **out of
scope for 0.5.4** and tracked separately as the 0.6.0 spec.

## Goals

- `pnpm provision repair` (or any re-provision) on an existing project must
  reuse the existing site-settings data source instead of creating a new
  one.
- The site-settings data source the scaffolder creates must carry a rich
  set of editable fields that let the operator control, end to end, the
  public surface of the generated site: SEO metadata, multi-level
  navigation (with dropdowns and a CTA), a multi-column footer with social
  links, and a theme (primary/accent colour + font).
- The generated `package.json` and `components/ui/` directory must always
  ship the same UI preset (the `site` / "Notion page builder" preset).
  The CLI must no longer ask the user which preset to use.
- All three changes ship as a single 0.5.4 patch release with tests.
- A user with an already-installed 0.5.3 project (e.g. the `digwis`
  testbench) can run `pnpm dlx @notionx/create-nextion-app@0.5.4 update`
  to pick up the changes and the project must keep working without
  re-creating the existing site-settings data source.

## Non-Goals

- No new published npm packages.
- No change to the runtime contract of `getSiteSettings()` for the four
  existing fields (name, tagline, description, default locale) — the new
  fields are additive.
- No change to the 4-step content-source provisioner contract (blog / page
  / etc.); this spec only touches `ensureSiteSettingsDatabase`.
- No rework of the multilingual routing. Even after 0.5.4, the runtime
  still treats the project as a single-locale site; the supported-locales
  list in `fallbackSiteConfig.locales` is informational only and does not
  affect which Notion data source the home page reads.
- No introduction of a new i18n system (no `next-intl`, no `react-intl`).
  Adding them is part of the 0.6.0 spec.

## Problem

### B — Site settings gets duplicated on every provision

`ensureSiteSettingsDatabase` in
`packages/create-nextion-app/src/provision/notion.ts` builds the data
source from scratch every time it is called. The content-source path
(`ensureNotionDatabase`) already handles reuse via two helpers:

- `findExistingDatabaseByStableKey` — search Notion for a data source
  whose description carries a `scaffoldKey: <key>` marker.
- `findExistingDatabaseByTitle` — fall back to a title match on the
  parent page.

`ensureSiteSettingsDatabase` does neither, so re-running provisioning
(e.g. via `nextion update` or `pnpm provision repair`) leaves a trail of
empty "Site Settings" data sources in the user's Notion workspace. After
two re-provisions the user has three empty tables and only the first one
is wired into `NOTION_SITE_SETTINGS_DATA_SOURCE_ID`.

### C — Site settings row is too thin to drive the site

`buildSiteSettingsSeedPage` writes four properties. `loadFromNotion` in
`lib/site/settings.ts` reads back those four properties plus a `Social
Image` URL. Everything else the public site renders — header navigation,
footer columns, accent colour, `<title>` and `<meta description>` on
every page — lives in `fallbackSiteConfig` in
`lib/site/config.ts` and can only be changed by editing code.

The user has been working around this by hand-editing the generated
project. That defeats the point of having Notion as the editor surface.

### E — The UI preset prompt is dead weight

`prompt.ts` exposes a `UI_PRESETS` table (minimal / site) and
`--ui-preset=<name>` CLI flag. Every shipped preset (blog, page) emits
the same answer: `site`. The `minimal` preset is only useful for users
who want to start with the bare primitives and `pnpm dlx shadcn add` the
rest themselves — and even then, none of the generated page-builder
components compile against it.

The prompt adds a question to every run, slows the interactive path, and
the only safe answer is "the same one we shipped last time". Removing
the question costs users nothing.

## Proposed Changes

### 0.5.4.A — Site settings reuse (problem B)

#### In `notion.ts`

Refactor `ensureSiteSettingsDatabase` to mirror the content-source flow:

1. Call `findExistingDatabaseByStableKey({ ..., stableKey: "site-settings" })`
   first. If it returns a match, use it (with a console warning if the
   workspace has more than one match, as the content-source path does).
2. Fall back to `findExistingDatabaseByTitle({ ..., title: "Site Settings" })`.
3. Only create a new data source if both lookups miss.
4. When creating, write the scaffold marker into the database
   description using the same pattern the content-source path uses: a
   description of the form
   `"[nextion-scaffold] key=site-settings\n<any user-provided text>"`
   and a call to `extractScaffoldKey(description)` to read it back. The
   constant `SCAFFOLD_MARKER_PREFIX = "[nextion-scaffold] key="` lives
   at the top of `notion.ts` (next to the rest of the scaffold markers)
   and is the single source of truth for the marker shape.
5. Add `reused: boolean` to `SiteSettingsProvisionResult`. The CLI summary
   line for site settings will then read either
   `♻️ Site Settings data source d13bd15c… (reused existing)` or
   `🆕 Site Settings data source d13bd15c… (created new)`.

#### In `notion.test.ts`

Add a unit test for `ensureSiteSettingsDatabase` that runs against a
stubbed `ntn api` response containing a pre-existing data source with
the `site-settings` stable key. Assert that the function returns the
existing id, marks `reused: true`, and does not call the data-source
create endpoint.

Add a second test that runs against a workspace with a same-titled data
source but no stable key, and asserts the title-match fallback is used
(also `reused: true`).

Add a third test for the cold-start case (nothing matches) that asserts
`reused: false` and that the create endpoint receives a description
containing the `scaffoldKey: site-settings` marker.

### 0.5.4.C — Rich site settings schema (problem C)

#### Notion schema additions

`buildSiteSettingsProperties` grows from 5 properties to **17** (the
existing 5 + 12 new). New properties are listed below; all of them are
additive so an existing site-settings data source from 0.5.3 continues
to work and just gets the new fields in the next re-provision (Notion
adds properties with no destructive effect on existing rows).

| Property | Notion type | Default seed value | Mapped to `SiteConfig` |
|---|---|---|---|
| **Meta Title** | rich_text | `{{projectName}}` | `seo.title` |
| **Meta Description** | rich_text | `{{projectName}}` | `seo.description` |
| **OG Image** | url | (empty) | `seo.ogImage` (falls back to `socialImageUrl`) |
| **Nav** | rich_text (JSON string) | `[{label:"Home",href:"/"},{label:"Blog",href:"/blog"}]` | `navigation.main` |
| **Nav CTA** | rich_text (JSON string) | `null` | `navigation.cta` |
| **Primary Color** | select (8 presets) | `slate` | `theme.primary` |
| **Accent Color** | select (8 presets) | `blue` | `theme.accent` |
| **Font Family** | select (3 presets) | `inter` | `theme.font` |
| **Footer Columns** | rich_text (JSON string) | `[]` | `footer.columns` |
| **Footer Copyright** | rich_text | `© {{year}} {{projectName}}` | `footer.copyright` |
| **Footer Social Links** | rich_text (JSON string) | `[]` | `footer.social` |
| **Footer Tagline** | rich_text | (empty) | `footer.tagline` |

The 8 colour presets for Primary/Accent are:
`slate, gray, zinc, red, orange, amber, green, blue`. The 3 font presets
are `inter, geist, system`. Storing them as `select` rather than
free-form keeps the editor experience fast and prevents typos from
breaking the theme pipeline.

The JSON-shaped properties (Nav, Nav CTA, Footer Columns, Footer Social
Links) ship as `rich_text` containing a single JSON-encoded string.
This is intentionally pragmatic: Notion's `rich_text` is the most
permissive type for storing a JSON blob, and the loader can parse it
defensively. (Switching to a Notion-API-native `multi_select` or
`relation` representation is a 0.7+ consideration; the JSON approach
matches what the rest of the scaffolder already does for navigation.)

#### `buildSiteSettingsSeedPage` update

`buildSiteSettingsSeedPage` writes a single Notion page with the
project name, a default description, and all 12 new properties seeded
to safe defaults. The current `databaseId` → `data_source_id` parent
fix from 0.5.3 is preserved unchanged.

#### `loadFromNotion` and `fallbackSiteConfig` updates

`fallbackSiteConfig` (in `templates/lib/site/config.ts.tmpl`) gains the
`seo`, `footer`, and `theme` fields with sensible static defaults, so
`SiteConfig` is a complete type after the change.

`loadFromNotion` (in `templates/lib/site/settings.ts.tmpl`) gains a
`readJson` helper that parses a `rich_text` property as JSON and
returns the default on any parse error. Each new field is read through
it (or `readRichText` / `readSelect` for the simpler ones) and merged
into the returned `SiteConfig`.

#### Render-side updates

- `templates/app/layout.tsx.tmpl` — `generateMetadata` reads
  `getSiteSettings()` and emits the `title`, `description`, and
  `openGraph.images` fields. The `<html data-theme="..." data-font="...">`
  attributes are set from the resolved theme so client components can
  pick them up.
- `templates/components/site/site-header.tsx.tmpl` — accepts the rich
  `navigation` shape (top-level items with optional `children` arrays
  rendered as dropdowns, plus an optional `cta` rendered as a button
  on the right). The previous "single link to the content source" is
  preserved as the default `main` seed.
- `templates/components/site/site-footer.tsx.tmpl` — rewritten to
  accept the `footer` shape. When `footer.columns` is empty (the
  default), the footer renders as a single copyright line; once the
  operator adds columns in Notion, the layout expands to a multi-column
  grid.
- A new tiny client component `templates/components/site/theme-bootstrap.tsx.tmpl`
  reads `document.documentElement.dataset.theme` / `data-font` and
  injects the matching CSS custom properties (the 8 colour presets are
  hard-coded in the file as `--theme-primary-50` through `--theme-primary-900`
  groups). This is *not* a full Tailwind theme switch; the component
  only sets a handful of CSS variables and the rendered site uses them
  via inline `style` or the existing `bg-primary` / `text-primary`
  tokens that already point at CSS variables.

#### Tests

- `notion.test.ts` gains a snapshot test for the new
  `buildSiteSettingsSeedPage` output covering all 17 properties.
- `notion.test.ts` gains tests for `readJson` (the new helper) — happy
  path, malformed JSON, non-string input.
- `metadata.test.ts` gains a test that asserts `getSiteSettings()`
  returns the new `seo` / `footer` / `theme` shape with sensible
  defaults when the Notion row is unreachable.
- The render tests gain a small assertion that the generated
  `app/layout.tsx` exports the right metadata keys and the
  `site-header.tsx` reads the new navigation shape.

### 0.5.4.E — Unify on the `site` UI preset (problem E)

#### Code changes

- `presets.ts` keeps `SITE_PRESET` as the single source of truth. The
  exported `MINIMAL_PRESET` constant is removed; the
  `normalizeUiPreset` helper becomes a no-op that just returns `"site"`.
- `prompt.ts` deletes the `UI_PRESETS` table and the `uiPreset`
  interactive prompt. `Answers` no longer carries `uiPreset`.
- `package.json.tmpl` deletes the `{{uiPreset}}` branch and always
  emits the `site` preset's components and `@radix-ui/*` dependencies.
- `templates/components/ui/*.tmpl` — unchanged. The `site` preset's
  file set is exactly the existing template directory, so no template
  file is deleted.
- CLI help text drops the `--ui-preset` flag.

#### Tests

- `presets.test.ts` deletes the `minimal` cases and pins the `site`
  component set as the only one the scaffolder may emit.
- `metadata.test.ts` removes the `uiPreset` test cases and replaces
  them with an assertion that `uiPreset` no longer appears on
  `Answers`.

#### Migration notes

- A user who somehow pinned `--ui-preset=minimal` on the CLI in a
  previous version will get a warning in the README that the flag has
  been removed and that their project now ships the `site` preset's
  components. The list of files in `components/ui/` will be different
  (larger) than what they had; the diff is purely additive and the
  `site` preset's components compile against the same Radix versions.
- The `package.json.tmpl`'s dependencies block already has
  `next-themes` and the Radix UI primitives the `site` preset needs;
  no new top-level dependencies are introduced by 0.5.4.

## Out of Scope (Tracked Separately)

The 0.6.0 spec covers the per-locale content-source split. That work
overlaps with 0.5.4 in two places and the boundary is:

- 0.5.4 *only* touches `ensureSiteSettingsDatabase`,
  `buildSiteSettingsProperties`, the corresponding `lib/site/*`
  templates, and the UI preset prompt. It does **not** touch
  `ensureNotionDatabase` or the content-source routes.
- 0.6.0 will add per-locale data sources for each content source, move
  the supported-locales list from `fallbackSiteConfig` into a runtime
  router, and add path-prefix routing (`/zh/blog/...`,
  `/en/blog/...`). It will *not* remove or reshape any of the 17
  site-settings properties that 0.5.4 introduces — the schema 0.5.4
  lands is a strict superset of what 0.6.0 needs.

The 0.6.0 spec is written separately and will be its own design
document under `docs/superpowers/specs/`.

## Testing Strategy

### Unit tests (Vitest)

Run on every push and on release workflow. 0.5.4 should land with all of
the following passing:

- Existing 56 tests in `packages/create-nextion-app/src/**` (all of
  which currently pass).
- New tests added by 0.5.4.A, 0.5.4.C, and 0.5.4.E (estimated total
  ~15 new test cases).

### Lint + typecheck

`pnpm lint` and `pnpm typecheck` at the workspace root must pass
cleanly. The new TypeScript types in `SiteConfig` and the `readJson`
helper are typed strictly — no `any` introduced.

## Rollout

### Order of operations

1. Land the code on `main` behind a single `0.5.4` changeset. Three
   `.changeset/*.md` files (one per sub-feature) merged into a single
   PR so the release workflow picks up exactly one bump.
2. Confirm the `release.yml` workflow runs end to end and the npm
   `latest` tag advances from `0.5.3` to `0.5.4`.
3. Update the README and the upgrading-nextion doc with a one-paragraph
   note about the schema additions (operators who hand-edit their site
   settings do not need to do anything — Notion preserves the new
   empty properties on next edit).
4. Hand off the testbench integration test results.

### Backwards compatibility

- **Existing 0.5.3 projects**: on first re-provision with 0.5.4 the
  existing site-settings data source is found by title match
  (B-fallback). Notion adds the 12 new properties automatically; the
  existing 4 fields keep their values. No operator action required.
- **CLI**: `--ui-preset=<name>` is removed in 0.5.4. The `parseArgs` layer
  drops the flag entirely, and any user still passing it gets a
  one-time `console.warn` on first invocation explaining that the flag
  has been removed and that the project now ships the `site` preset's
  components. No 0.5.5 follow-up is planned — the flag is gone.
- **Runtime**: `getSiteSettings()` callers that destructure the
  existing four fields continue to work unchanged. Callers that
  destructure `fallbackSiteConfig` directly (none in the project, but
  any third-party copying the template) get new top-level keys —
  additive, non-breaking.

## Open Questions

- None for 0.5.4. The 0.6.0 spec will revisit slug alignment across
  locales and the i18n resource store (JSON vs Notion vs something
  else) — neither is a 0.5.4 concern.
